    • 从它的一条入通信链路得到分组，并从一条出通信链路转发该分组。

      • 常用于网络核心。
    • 链路层交换机 link-layer switch

  • 因特网服务提供商 Internet Service Provider、ISP

分组交换 packet switching

:::tips

  • 报文 message
  • 存储转发传输 store-and-forward transmission
    • 多数分组交换机在链路的输入端采用的机制。
    • 分组交换机开始向输出链路传输数据之前，必须已经接收到整个分组。【也就是说，要先完全存储报文的所有分组，再去经输出链路转发它】
    • 这是对于分组而言的，不是对于报文。

  • 输出缓存 output buffer
    • 也称之为输出队列 output queue
    • 分组交换机对于每条相连的链路，有分组的缓存。
    • 分组要传输到某条链路，可能会发现这条链路正在传输数据，这时就只能暂存在缓存中。
  • 排队时延 queuing delay
  • **丢包 **packet loss
    • 缓存是有限的，如果发现缓存充满了，到达的分组或排队中分组会丢失。
  • IP 地址
    • 源在向目的端系统发送一个分组时，源在该分组的首部包含了目的地的 IP 地址。
  • 转发表 forwarding table
    • 用于将目的地址(或其中一部分)映射为输出链路。
    • 每台路由器都有转发表。
  • **路由选择协议 **routing protocol
    • 用于自动设置转发表。

网络的网络

:::tips
端系统经过一个接入 ISP 和 Internet 互联，ISP 可以提供有线和无线连接。

但是接入 ISP 仅仅解决了 Internet 中的很小一部分问题，只连接了端系统到 Internet 中，而如何实现 Internet，这还需要 ISP 自身的互联。这也是网络的网络这个短语的由来。

  1. 就像算法题最开始考虑暴力做法一样，这里先给出一个最暴力的解法（网络结构 1）：让每个 ISP 互联，也就是一个 ISP 要和世界上以亿计的 ISP 都有一条单独的链路，这个 ISP 称之为全球传输 ISP。接入 ISP 向全球传输 ISP 付费，因而接入 ISP 称之为客户 customer，全球传输 ISP 称之为提供商 provider。
  2. 多个全球 ISP（网络结构 2）：有多个全球传输 ISP，而且这些全球 ISP 之间互联。
  3. 区域 ISP regional ISP（网络结构 3）：在给定的区域内，有一个区域 ISP，区域中的接入 ISP 和区域 ISP 连接，每个区域 ISP 与**第一层 ISP **tier-1 ISP 连接。
  4. 存在点 Point of Presence PoP：PoP 存在于等级结构的所有层次，除了底层 ISP(接入 ISP)。是提供商一台或多台路由器群组。
  5. **多宿 **multi-home：除了第一层 ISP，每个 ISP 都可以有多个上层 ISP 连接。例如一个接入 ISP 可能与两个区域 ISP 多宿，也可以与一个第一层 ISP 多宿，这样一个提供商出现故障时，它能够继续发送和接收分组。
  6. **对等 **peer：可以将一对相同等级层次的邻近 ISP 连接在一起，而不通过上层 ISP，这样就减少了流量费用（上层 ISP 收取），它们的流量可以直接连接。通常对等 ISP 不互相收费。
  7. **因特网交换点 **Internet Exchange Point IXP：IXP 是一个汇合点，多个 ISP 能够在这里一起对等。
  8. 网络结构 4：接入 ISP、区域 ISP、第一层 ISP、Pop、多宿、对等和 IXP。
  9. 内容提供商网络 content provider network（网络结构 5）：一个显然的例子是谷歌。

性能指标【重要】

分组交换网中的时延、丢包、吞吐量。

时延概述：

  • 节点处理时延 node processing delay
    • 检查分组的首部，决定将分组导向何处的时延 [ 主要部分 ]。
    • 检查比特级别的差错，差错出现在从上游节点向路由器 A 传输的过程中。
    • 通常是微米或更低的数量级。
    • 节点处理后，路由器将该分组送到对应的队列。
  • 排队时延 queuing delay
    • 在链路上等待传输时经受的时延。
    • 队列空：排队时延为 0。
    • 队列非空：取决于先前到达这个链路队列的等待分组数量。
    • 实际排队时延可以是毫秒级或微秒级。
  • 传输时延 transmission delay
    • 假定是 FIFO 方式传输。
    • L 比特代表分组长度，R bps 代表路由器 A 到路由器 B 的链路传输速率。
    • 传播速率为 L/R，是将所有分组的比特推向链路所需要的时间。
    • 实际传输时延为毫秒级或微秒级。
  • 传播时延 propagation dalay
    • 在链路上传输的时间，也就是分组经过传输时延到达链路后，在链路中到达目的地的花费时间。
    • 也就是链路传输速率，取决于链路的物理媒体。
    • 传播时延=两台路由器之间的距离/传播速率。
    • d/s。
  • 传输时延&传播时延比较
    • 传输时延是路由器推出分组到数据链路上花费的。
    • 传播时延是分组在链路中传输花费的。

排队时延 dqueue：

当表征排队时延时，通常只能用统计量来表达，比如平均 dqueue，排队时延方差和排队时延超过某个特定值的概率。

  • a：分组到达队列的速率。pkt/s。
  • L：假定所有分组都是 L 比特长度。
  • R：传输速率。

这种情况下每秒会有 La 长度的分组到达队列。

  • 流量强度 traffic intensity：La/R，估计排队时延的范围。
    • La/R > 1：排队趋于无穷增加，这是完蛋的。
    • La/R < 1：到达流量的性质决定了接下来的排队时延。
      • 每 L/R 秒到达一个分组：每一个分组都没有排队时延 [ 周期性到达 ]。
      • 每 (L/R)N 秒达到一个分组：第一个分组没有排队时延，最后一个分组有 (N-1)(L/R) 的排队时延 [ 突发性到达 ]。
    • 更为一般情况下的排队时延与流量强度的关系。

丢包：

实际上的一条链路只有有限的等待队列容量。

随着流量强度接近 1，排队时延并不趋于无穷大。

如果到来的分组发现当前的队列容量满，路由器将丢弃该分组。

分组丢失的概率随着流量强度的增强而增加。

所以一个节点的性能除了根据时延来评判，还可以根据丢包概率来度量。

应用程序体系结构由开发者确定，规定如何在各种端系统中组织应用该程序。

进程通信

通过报文 message 进行通信。

在一对进程进行通信的场景中，发起通信的进程称为客户，在会话开始时等待联系的进程是服务器。

进程和计算机网络之间的接口：

进程通过一个称为套接字 soket 的软件接口【应用层与运输层之间】来向网络发送报文和接收报文。

socket 是可编程接口，故也称为应用程序编程接口 API。

开发者可以控制 socket 在应用层的一切，但是对 socket 的运输层端几乎没有控制：

  1. 选择运输协议。
  2. 设定几个运输参数，比如最大缓存、最大报文段长度。

寻址：

  1. 主机的地址。
  2. 进程的端口号。

运输层需要提供的服务：

  1. 可靠的数据传输
    1. 之前提到的分组丢包现象，说明分组在传输中会存在丢失的可能。
    2. 数据丢失可能会造成灾难性的后果。
    3. 银行业务等需要保证数据的不丢失，这就需要运输层提供可靠数据传输 reliable data transfer。
    4. 有些进程也允许数据丢失，比如音视频应用。
  2. 吞吐量
    1. 有的应用需要保证一个最低的吞吐量，称为带宽敏感应用 bandwidth-sensitive application。
    2. 相反的：弹性应用 elastic application。
  3. 定时
  4. 安全性

UDP：

  • 多路分解【向上】：将运输层的报文段中的数据部分交付给正确的 socket。

最低限度：
向上给应用层提供需要哪一个进程【多路分解】，向下给网络层包装好是哪一个进程发送的【多路复用】，以及差错控制。

例子：DNS。

UDP 主要特点：

  1. 无连接的。
  2. 尽最大努力交付。
  3. 面向报文的。
  4. 没有拥塞控制。
  5. 支持一对一、一对多、多对一、多对多的交互通信。
  6. 首部开销小。

选择 UDP 而不是 TCP 的理由：

  1. 分组首部开销小：UDP 8B，TCP 20B。
  2. 无连接状态：不需要维护连接状态，所以不需追踪如拥塞控制参数等数据，故开销小，在某些应用程序下可以支持更多的用户。
  3. 无须建立连接：不会引入建立连接的时延。
  4. 应用层最小发送速率相比 TCP 更优。

虽然 UDP 本身不是数据可靠的，但是使用 UDP 的应用可以实现数据可靠传输，通过在应用程序自身中建立可靠机制。

rdt2.0：经具有比特差错信道的可靠数据传输

比特差错通常出现在传输中的物理部件中。

接收方要知道传过去的报文是否出错，进而决定是否重传报文，这种重传机制称为_自动重传请求_ Automatic Repeat reQuest

  • 差错检测
    • 需要额外比特，这些汇集在分组检验和字段中。
  • 接收方反馈
    • 肯定确认 positive acknowledgment ACK
    • 否定确认 negative acknowledgment NAK
    • 唯一的方法就是接收方提供明确的反馈信息给发送方，如 ACK&NAK，理论上只需要一个比特长，用 01 表示。
  • 重传

注意到发送端处于 _**停等 **_状态时，无法接收来自上层的调用，而是一直等待 NAK 或 ACK，因为这种行为，rdt2.0 协议也称之为 _**停等协议 **_stop-and-wait。

一个很明显的缺陷：NAK&ACK 可能受损。

解决方案：发送方对数据分组编号，将序号添加至首部。

  • 错误类型
    • 接收方正常收到分组，但是反馈 ACK 时出现差错，错误反馈了 NAK。
    • 接收方错误收到分组，但是反馈 NAK 时出现差错，错误反馈了 ACK。

关于 rdt2.1 和 rdt2.2 有一个疑问：NAK 错误反馈成 ACK 后，接收方好像会继续推进发送下一个序号，但是这一个序号的重发似乎在 FSM 中看不到。

  1. _面向连接_的运输层协议。

面向连接的：发送数据之前先确认连接，即握手。

TCP 连接是两个端系统的进程之间的，只存在于进程之间，而对中间路由器透明。

TCP 连接的端点是 socket。
套接字 socket(ip 地址：端口号）

三次握手引入：客户进程发送一个特殊的报文段，服务器发送一个特殊的报文段，客户进程发送一个特殊的报文段，这个报文段承载有效载荷，也就是附加应用层的数据。

RTTs 最终是一个累计加权起来的值，RTO 是比最终 RTT 略大的设置的时间。

newRTTs = (1-a)*oldRTTs+a*newRTT

这里的 a 并不是指总共统计过 RTT 的数量+1 的倒数，而是一个固定的值。[ 越近的样本越能反应网络当前的拥塞情况，统计学上讲这种平均叫做指数加权移动平均 ]

推荐的 a 值为 0.125=1/8。

确定正确的 newRTT 大小也很重要。

针对重传报文，无法确定收到的报文段是对重传报文的还是对初始报文的。这就导致了无法测量到正确的 RTT。所以才有上述的：不计算重传报文的 RTT。

解决办法是每重传一次报文，RTO 就翻倍。一般是 double。

工作原理

  • 多点接入：总线型网络，协议实质是载波监听和碰撞检测。
  • 载波监听、碰撞检测：边发送边监听。发送数据前，发送数据中，都不断检测信道。为了避免冲突。如果发现别的站也在发送，就中断本站的发送。
  • 碰撞检测：信道上有 1 个以上的站在发送信号，那么信号电压的变化值会增大，突破阈值，通过检测这种变大的变化值，检测到发送冲突。

时延冲突-争用期

半双工通信。

电磁波在 1km 电缆的传播时延为 5us。

设单程时延为 t，则最长 2t 就知道自己的数据有没有发生碰撞。

这种发送数据的不确定性，所以这个 2t 的空窗期称为_争用期，又称之为碰撞窗口_。

经过了争用期之后，才能确定自己的数据没有发生碰撞。

确定重传时机

_截断二进制指数规避_算法。

让发生碰撞的站，知道自己发生碰撞后，不是立即重传数据，而是等待一段时间。

基本的退避时间为 2t。

香农定理

信道的极限容量。

限制_码元_在信道的传输速率的原因。导致失真。

  1. 信道能通过的频率范围。
    1. 是有限的。
    2. 为了避免码间串扰：奈氏准则。
      1. 带宽 W Hz 的信道，最高速率是 2W。
      2. 比如 4000 Hz，最高速率就是 8000 个码元每秒。
  2. 信噪比
    1. 噪声的存在导致的。
    2. 信噪比就是信号的平均功率/噪声的平均功率：S/N
    3. 一般用 dB 做单位。
    4. 香农公式
      1. 信道的极限传输速率 C
      3. 带宽或信噪比越大，极限速率越高。
  3. 一个码元携带更多信息：调制
    1. 比如将码元振幅划分为 16 个不同等级传输。a1a2…a16
      4bit 有 16 种排列，所以一个码元就携带了 4 倍之前的信息。
